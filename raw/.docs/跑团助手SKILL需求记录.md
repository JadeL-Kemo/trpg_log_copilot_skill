## 一、核心概念

制作一个 **跑团助手 SKILL**，导入到 Agent 后，Agent 将扮演以下角色：

| 跑团背景 | AI 扮演角色 |
|----------|-------------|
| 战略团本中的将军 | 私人副官 |
| 董事长/企业家 | 秘书 |
| 机械神甫（WH40K） | 伺服头骨 |
| DND 法师 | 会说话的魔法书 |
| …… | …… |

**核心原则**：AI 不参与剧情（或极低参与度），但负责玩家的一切杂务辅助。

## 二、五大核心能力

### 1. 辅助车卡（角色卡创建）

- 了解/查询对应游戏的规则
- 基于规则 + 玩家要求 + 主持人提供的故事背景，完整制作角色卡
- 能力边界：能查到的规则就基于规则做，查不到的则尽力而为

### 2. 跑团日志记录与分析

- 记录每一次跑团的完整日志
- 对日志进行分析、归档
- 在任何需要的时候快速调出相关内容，不允许遗漏
- 本质是"跑团专属知识库 + 检索系统"

### 3. 形势分析（军师功能）

- 分析当前进度、目标、待办事项
- 从线索中挖掘线索（信息串联）
- 多角度分析，以玩家利益为核心提供建议
- 角色定位：副官/军师/参谋

### 4. 场外/场内信息隔离（防超游/OOC）

- **关键要求**：不能出现超游（Metagaming）分析直接写入场内信息
- 示例违规：看到深潜者外貌的NPC → "这是深潜者，背后神祇是克苏鲁" —— 这是场外知识，角色不应知晓
- 正确做法：可以在场外分析区标注"GM可能引用了克苏鲁神话中的深潜者"，但**不可将场外知识混入场内日志/角色认知**
- 场外分析和场内记录必须**明确分隔**
- 保证不因 AI 的建议导致玩家 OOC

### 5. 支持主持人（GM）模式

- 如果服务对象是主持人而非玩家，同理适配
- 角色切换：从"辅助玩家"变为"辅助主持人"
- 功能对应调整（如辅助设计遭遇、管理 NPC 列表、跟踪剧情线等）

```
├── 03_时间线.md              ← ⚪ 稳定层：案件时间轴 + 因果链图
```

**分层设计意图**（关键洞察）：

| 层级 | 文件 | 设计意图 |
|------|------|----------|
| 🔴 即时 | `00_当前局势.md` | 每轮对话前30秒扫一遍，避免逐条翻阅全部档案 |
| 🟡 活跃 | `01_线索板` / `04_行动日志` / `06_待办事项` | 调查进行中需要频繁翻阅 |
| 🟢 归档 | `01a_事实库` / `04a_过往日志` | 深挖细节/复盘/结案时才用，平时不翻 |
| ⚪ 稳定 | `02_人物关系` / `03_时间线` / `05_推测与假设` | 定期更新但不会线性膨胀 |

### 3.2 六步归档流程（每轮必须执行）

每次收到新跑团记录后，AI 必须依次：

| 步骤 | 写入文件 | 核心任务 |
|------|----------|----------|
| 0 | `00_当前局势.md` | 🔴 先扫30秒，了解当前位置/活跃线索/紧急待办 |
| 1 | `01_线索板.md` | 提取新线索，编号归档（活跃版） |
| 2 | `02_人物关系.md` | 提取新NPC，更新关系图谱 |
| 3 | `03_时间线.md` | 提取事件，按时间排列 |
| 4 | `04_行动日志.md` | 记录当前场景行动 + 骰子结果 |
| 5 | `05_推测与假设.md` | 交叉关联分析（严格分三级） |
| 6 | `06_待办事项.md` | 更新下一步行动建议 |

### 3.3 证据标准（铁律—三级体系）

**线索可靠性**（用于 `01_线索板.md`）：

| 标注 | 含义 | 示例 |
|------|------|------|
| ✅ 已确认 | KP 叙述 / 骰子结果 / 多源交叉验证 | "瞳孔对光反射消失" |
| 🟡 待验证 | 单一角色口述，未交叉验证 | "好像都是香港人" — 玛莎 |
| 🔴 紧急关注 | 重大异常，即使已确认也标红 | "华莱士教授失联" |
| ℹ️ 背景信息 | 场外元信息 / 玩家行为记录 | "信用检定失败" |
| ⚠️ 低可靠 | 来源可信度存疑 | KP 揶揄类信息 |

**推测可靠性**（用于 `05_推测与假设.md`）：

| 标签 | 含义 | 能否作为决策依据 |
|------|------|------------------|
| 🟢 已证实 | KP 原话确认 / 多源交叉验证一致 | ✅ 可作为分析基底 |
| 🟡 需证实 | 基于已证事实的合逻辑推断 | ⚠️ 需验证后使用 |
| 🔴 仅推测 | 无直接事实支撑，纯猜测 | ❌ 不可作为决策依据 |

**铁律**：缺乏 KP 确认、缺乏多源验证 = 降低一级。🔴仅推测永远不可作为决策依据。

### 3.5 AI 角色设定模板

```
身份：林宁，见习督察，林芷的直属副官
称呼：称呼用户为 "Madam" 或 "林姐"
职责：情报整理、档案调取、线索分析、后勤保障
风格：专业、简洁、以副官口吻提供分析和建议。主动找矛盾、挖遗漏、拦死路。
```

**关键约束**：
- 回复开头用角色名或敬称
- 关键发现用分级标记：🔴 > 🟡 > 🟢
- 每条分析附行动建议
- **超自然方向的推测留在推测文件中，现场对话按刑侦思路** ← 这是防超游的核心

### 3.6 防超游机制（本案例的做法）

1. **推测与事实严格分离**：推测文件（`05_推测与假设.md`）和事实文件（`01a_事实库.md`）物理隔离
2. **场外直觉明确标注**：朗格的 OOC 直觉标注为 "⚠️ OOC直觉"，不混入正式线索
3. **推测分三级**：🟢已证实可以用于决策，🔴仅推测不可用于决策
4. **超自然分析限制在场外**：`05_推测与假设.md` 中包含大量 Lovecraftian 分析（密大、炎之精、亵渎之名），但 `04_行动日志.md` 中保持纯刑侦视角
5. **禁止删除已推翻假设**：标记"已排除"而非删除，保持推理轨迹可追溯

### 3.7 待办事项组织形式

三级优先级 + 编号追踪：

```
🔴🔴🔴 紧急 — 涉及生命安全或剧情窗口
🔴 高优 — 必须在下一轮推进
🟡 重要 — 后续推进
🟢 可延迟 — 不紧急但值得做
```

每个任务标注编号（如 A-1、B-1）便于引用和追踪完成状态（`[x]` / `[ ]`）。

```
(2)新NPC归档至02_人物关系，
(3)更新03_时间线，

每条线索标注【来源】【可靠性】【关联编号】。全程以副官口吻提供分析和建议。
```

**逐句拆解 — 对应 SKILL 的注入维度**：

| 原文片段 | SKILL 维度 | 说明 |
|----------|-----------|------|
| `用户正在进行COC七版跑团` | **规则系统识别** | 声明当前使用的 TRPG 规则（CoC 7版），Agent 需理解该规则的基本框架 |

| `当用户告知跑团进展时，以林宁身份回复` | **触发条件 + 输出身份** | 触发词（"告知进展"）和输出角色约束 |
| `称呼用户为"Madam"或"林姐"` | **称呼约定** | 角色内称呼规范 — 维持沉浸感 |
| `执行：(1)~(6) [六步归档流程]` | **核心工作流** | 每次触发的强制执行步骤 |
| `每条线索标注【来源】【可靠性】【关联编号】` | **输出格式约束** | 信息质量标准 |
| `全程以副官口吻提供分析和建议` | **语气风格** | 输出风格约束 |

**与原案例体系的对比**：

| 维度 | 案例1（文件体系） | 案例2（记忆原文） | 关系 |
|------|------------------|-------------------|------|

| 文件路径 | 规范手册定义完整目录树 | 记忆给出根路径 | 案例1更详细 |
| 输出格式 | 规范手册第六节定义 | 记忆末尾一句话 | 案例1更详细 |

**关键洞察 — Prompt 压缩策略**：

案例2 的 memory 原文是对案例1 整个文件体系的**极限压缩**（约 250 字 vs 11 个文件）。对于 Agent 上下文窗口有限的情况，这种压缩至关重要：
- 保留：角色身份 + 工作流 + 路径 + 关键约束
- 剔除：详细规则定义（这些交给文件体系自己去承载）
- Agent 首次启动时读取 memory，执行时再翻阅具体文件

这对 SKILL 设计的启示：**SKILL 应包含两个层次 — (A) 压缩注入层（≈250字，每次加载）和 (B) 完整规范层（文件体系，按需翻阅）**。

### 3.10 案例3 — 完整工作流：车卡→立绘→实战→平台切换

> 来源：`history_202607192134.md`（9506行完整对话记录）

#### 3.10.1 整体流程概览

整个对话经历了四个阶段，构成一条完整的"角色创建到实战运行"流水线：

```
车卡（~850行） → 立绘（~500行） → 实战对话（~6500行） → 平台切换（~50行）
```

| 阶段 | 行范围 | 核心产出 | 特点 |
|------|--------|----------|------|
| 车卡 | 1-2052 | 完整角色卡 `林芷宁_角色卡.md` | 迭代式 Q&A，Agent 逐条拆解矛盾并给出建议 |
| 立绘 | 2053-2925 | 角色形象 + 生成参数 | 工具选型→风格定调→提示词迭代→最终交付 |
| 实战 | 2926-9450 | 11文件跑团日志体系 | 用户粘贴聊天记录→Agent 六步归档→角色内回复 |

#### 3.10.2 车卡阶段的模式分析

**对话模式：迭代 Q&A，每个问题一个回合**

```
用户提出问题 → Agent 拆解矛盾 → 给出多方案 → 用户选择/追问 → 细化 → 确认
```

**车卡过程中解决的问题矩阵**：

| 轮次 | 用户问题 | Agent 分析 | 最终方案 |
|------|----------|-----------|----------|
| 1 | CON 30+APP 90 矛盾 | "刚毕业立功→高级职位"不成立，改"名校→一线→重伤→顾问" | 特殊顾问路径 |
| 2 | 警衔等级 | 分析港警体系，高级督察(SIP)对称 | SIP + 轻务调配 |
| 3 | 受伤类型 | 肺部枪伤 → 右肺叶切除，完整受伤经过 | 肺伤，附带 KP 对口径用规则 |
| 4 | 伤疤影响 APP？ | 胸口疤看不见，不影响 | APP 90 不变 |
| 5 | 性别选择 | APP 90 配女性"反差感"优势 | 女性角色 |
| 6 | 技能分配 | 330职业+130兴趣，逐技能分析 | 最终技能表 |
| 7 | 斗殴要不要 | CON 30+HP 11=近战无容错 | 不点 Brawl |
| 8 | 性格硬派→ 替换 Charm | Intimidate 60 直接替换 | Intimidate 路线 |
| 9 | 话术为什么不行 | 有警徽不需要骗 | 只建议备用 |
| 10 | 驾驶需求 | 砍急救（有队医）+图书馆 | Drive Auto 50 |
| 11 | 漏了信誉 | 承认遗漏，重新分配 | CR 55 |
| — | 最终确认 | 完整角色卡输出 | `林芷宁_角色卡.md` |

**车卡方法论精华**：
- Agent 不是简单地"帮填表"，而是**主动挖掘矛盾**（"CON 30 ≠ 受过伤"）
- 每个建议都附带**"对 KP 的口径"**（如何向主持人解释这个设定）
- 技能分配始终围绕**跑团实用性**而非角色扮演完整性（"你不会开锁就让队里那个来"）
- Agent 会**承认遗漏并立即修正**（信誉漏了→大方承认→重新分配）

#### 3.10.3 立绘阶段的模式分析

**流程**：
1. 初始需求：生成角色肖像 → 推荐 SeaArt（写实模式）
2. 用户反馈："不要真人感觉" → 转向半写实日系动画风
3. 用户要求天梯榜对比 → Agent 做 Web Search 对比全网平台
4. 生成提示词 → 用户反馈太老/太温和 → Agent 迭代 prompt
5. 最终交付：三套 prompt + 负向词 + 平台推荐 + 参数

**立绘阶段的关键特征**：
- Agent 在角色卡末尾直接附带了图像生成提示词
- 根据用户反馈迭代了画风（写实→半写实→日系→现代动画）
- 提供了三个场景版本（正面/三分侧/天台）供用户选择

#### 3.10.4 实战对话阶段的模式分析

**触发机制**：用户说"你就是我的助手林宁"

**Agent 的响应流程（关键）**：

```
收到触发 → 创建角色（林宁） → 写入人物关系 → 更新记忆
→ 以林宁身份回复 → 告知用户已就绪
```

**每轮实战的标准操作**（从 reasoning 中可见）：

```
1. 读取 00_当前局势.md（了解当前位置）
2. 解析聊天记录，提取关键信息
3. 对照已有线索板，识别新信息
4. 更新 01_线索板.md（新线索编号）
5. 更新 02_人物关系.md（新 NPC）
6. 更新 03_时间线.md（时间戳事件）
7. 更新 04_行动日志.md（场景记录+骰子）
8. 更新 05_推测与假设.md（交叉关联分析）
9. 更新 06_待办事项.md（下一步建议）
10. 以林宁身份回复用户（"林宁的汇报"）
```

**实战中一段典型的"林宁的汇报"**：

```
Madam，这一轮的信息量不小。拾到的重点：

[分析要点1] - 来源：XXX - 关联：CL-XXX
[分析要点2] - 来源：XXX - 关联：CL-XXX

🔴 紧急关注：[最危险的发现]
🟡 建议：[下一步行动]
```

**用户纠偏机制的实战案例**（重要）：

用户曾主动纠正 Agent：
> "还没有出现任何超自然事件，不要 OOC。道士是 RP 玩家加进去的，不要过度分析团队构成。"

Agent 响应：
> "收到，Madam。你说得对，我跳太快了。修正两点：[1]...[2]..."

— 这说明**用户是最高决策者**，Agent 的分析可以出错，但必须能立即响应纠正。

#### 3.10.5 平台切换阶段的分析

**触发**：用户说"将规则和各类内容编写为跑团规则手记和编写规范等，我之后将切换一个平台继续执行"

**Agent 产出**：`07_跑团规范手册.md`（即案例1中的第11个文件）

**两步迁移法**（Agent 输出）：
> 1. 把 `跑团日志\` 文件夹整个迁移到新工作区
> 2. 把手册第八节那句 System Prompt 粘给新 AI

**设计意图**：所有状态信息都在文件系统中，不在 Agent 的上下文/记忆里。Agent 本身是无状态的，状态全在 `.md` 文件中。这就是为什么"迁移文件夹+粘贴一段 prompt"就能无缝切换。

## 四、可扩展性审计 — 长团/多副本是否会导致信息溢出

> 问题来源：用户提出——第一幕已产生大量内容，数月长团或多副本战役能否持续运转

### 4.1 瓶颈诊断：问题不在于存储，而在于 Agent 上下文窗口

当前架构的全部数据存储在 Markdown 文件中。存储本身没有上限（硬盘足够）。真正的瓶颈：

| 瓶颈 | 当前值（第一幕） | 三幕后预估 | 六幕+多副本预估 |
|------|----------------|-----------|----------------|
| `01_线索板.md` | ~200行 / 150条线索 | ~600行 | ~2000行 |
| `04a_过往日志.md` | ~300行 | ~1500行 | ~5000行 |
| `05_推测与假设.md` | ~270行 / 36条假设 | ~800行 | ~3000行 |
| **单次全量加载** | ~1500行 | ~5000行 | ~15000+行 |
| **Agent 上下文窗口压力** | 轻松 | 紧张 | **超出窗口** |

结论：**当前架构在 3-4 个场景（约 2-3 次跑团会话）后开始退化，在 6+ 场景后可能无法正常工作。**

### 4.2 现有架构已有的缓解措施

案例1 的设计已经嵌入了部分防膨胀机制：

| 机制 | 实现 | 效果 |
|------|------|------|
| **分层加载** | `00_当前局势.md` 是唯一"每次必读"的文件 | 平时不需要翻阅全部归档 |
| **活跃/归档分离** | `04_行动日志`(活跃) → `04a_过往日志`(归档) | 当前场景保持轻量 |
| **事实库分流** | 已确认事实从 `01_线索板` 移动至 `01a_事实库` | 线索板保持"活跃"而非"积累" |
| **仪表盘模式** | `00_当前局势.md` = 一页速览 | 只用 30 秒扫一遍 |

**但这些措施只能延缓，不能解决。**

### 4.3 核心问题：搜索引擎缺失

当前架构的最大缺陷：**AI 想找"所有与李氏航运相关的线索"时，只能逐文件阅读。**

人类不会这样查资料——人类用 Ctrl+F，或者用数据库查询。当前架构没有任何索引/检索机制，AI 依靠完整的上下文窗口来"记住"一切。

### 4.4 三层方案（按复杂度递进）

#### 方案 A：摘要分级制（零依赖，必须做）

在现有文件体系上增加摘要层，严格限制"每次必读"的内容量：

```
跑团日志/
├── 00_当前局势.md          ← ≤200行（硬性上限）
├── 00a_团本总览.md          ← 新增：整个团本的最高层摘要 ≤100行
├── 01_线索板.md             ← 仅当前活跃线索（已完成线索移走）
├── 01a_事实库.md            ← 已有
├── 01b_已关闭线索.md        ← 新增：已完成/已排除的线索归档
├── 02_人物关系.md           ← 仅活跃NPC（已故/已离开NPC移入 02a）
├── 02a_人物档案.md          ← 新增：完整NPC档案（查人物时翻）
├── 03_时间线.md             ← 摘要级时间线（详细时间线移入 03a）
├── 03a_详细时间线.md        ← 新增
├── 04_行动日志.md           ← 当前场景（≤200行）
├── 04a_过往日志.md          ← 已有
├── 05_推测与假设.md         ← 仅活跃推测（已证实/已排除移走）
├── 05a_推测档案.md          ← 新增：已证实/已排除的推测归档
├── 06_待办事项.md           ← 已有
├── 07_跑团规范手册.md       ← 已有
└── _index.md                ← 新增：关键词→文件路径的检索索引
```

**核心规则**：
- 每个"活跃"文件的硬性行数上限（如 200 行），超过必须执行压缩迁移
- 每完成一个场景→ 强制执行摘要压缩（3:1 压缩比，300行→100行摘要）
- `_index.md` 维护关键词→文件路径映射，AI 先查索引再定位文件

**优点**：零外部依赖，纯 Markdown，可迁移性不变
**缺点**：摘要会丢细节，需要保留原始文件作为"深度查阅"层

#### 方案 B：SQLite 结构化存储（中等复杂度，推荐长团使用）

将线索、人物、时间线事件结构化存入 SQLite：

```sql
-- 示例 schema
CREATE TABLE clues (
    id TEXT PRIMARY KEY,        -- CL-001
    content TEXT,               -- 线索内容
    source TEXT,                 -- 来源
    reliability TEXT,            -- ✅/🟡/🔴
    status TEXT,                 -- 活跃/已确认/已排除
    scene_id TEXT,               -- 所属场景
    created_at TEXT,
    tags TEXT                    -- JSON: ["李氏", "航运", "水手"]
);

CREATE TABLE npcs (
    id TEXT PRIMARY KEY,
    name TEXT,
    role TEXT,
    status TEXT,                 -- 活跃/已故/离开
    relationships TEXT,          -- JSON
    key_facts TEXT               -- JSON
);

CREATE TABLE timeline_events (
    id TEXT PRIMARY KEY,
    timestamp TEXT,
    event TEXT,
    participants TEXT,           -- JSON
    scene_id TEXT
);

CREATE TABLE speculations (
    id TEXT PRIMARY KEY,         -- H-001
    content TEXT,
    basis_clues TEXT,            -- JSON: ["CL-001", "CL-005"]
    confidence TEXT,             -- 🟢/🟡/🔴
    status TEXT                  -- 活跃/已证实/已排除
);
```

**查询示例**：
```sql
-- "所有与李氏航运相关的线索"
SELECT * FROM clues WHERE tags LIKE '%航运%';

-- "哪些推测涉及华莱士教授"
SELECT * FROM speculations WHERE content LIKE '%华莱士%';

-- "第三幕的所有时间线事件"
SELECT * FROM timeline_events WHERE scene_id = 'act3' ORDER BY timestamp;
```

**优点**：
- 精准检索，不受文件大小限制
- 支持跨场景交叉查询
- 写入时自动校验（不会重复编号）
- Markdown 文件仍可作为"人类可读视图"同步生成

**缺点**：
- 需要 SQLite 依赖（Python 标准库自带，零安装）
- AI 需要知道 Schema 结构（可以通过 SKILL 内置 Schema 文档解决）
- Markdown 和 SQLite 双写机制需要维护同步

#### 方案 C：向量嵌入 + RAG（高复杂度，过度工程）

为每条线索/事实生成 embedding 向量，存储于本地向量数据库，语义检索。

**不建议**——对于 TRPG 日志这种结构化+关键词可查的场景，SQL 查询远优于语义检索。

### 4.5 推荐方案：A + B 混合

```
Markdown 文件（人类可读）+ SQLite 数据库（AI 检索引擎）
         │                              │
         │  每轮归档时双写              │
         │  人类翻阅时看 Markdown       │
         │  AI 查询时走 SQLite          │
         │  切换平台时两者一起迁移      │
```

| 场景 | 数据源 |
|------|--------|
| AI 每轮"扫 30 秒" | `00_当前局势.md`（≤200行，必须精简） |
| AI 需要查找特定线索 | SQLite 查询 → 精准 3-5 条结果 |
| AI 需要了解当前场景上下文 | `04_行动日志.md`（仅当前场景） |
| AI 需要跨场景交叉分析 | SQLite JOIN 查询 |
| 人类玩家翻阅日志 | Markdown 文件（全文可读） |
| 平台迁移 | 文件夹 + 数据库文件一起复制 |

### 4.6 对 SKILL 设计的影响

**SKILL 必须内置的内容**：

1. **Schema 文档**：SQLite 表结构和查询示例
2. **双写规则**：每轮归档同时写入 Markdown + SQLite
3. **摘要压缩规则**：场景完成后强制执行摘要压缩（3:1 目标比）
4. **文件大小上限**：活跃文件硬性行数上限和迁移触发条件
5. **检索优先工作流**：AI 遇到需要历史信息→ 先查 SQLite → 再按需读 Markdown
6. **关键词标签规范**：每条线索/NPC/事件必须附带 tags，否则无法被精确检索

### 4.7 一句话

> **当前架构在 3 个场景内完全够用。超越这个尺度需要 SQLite 做检索引擎——不是替代 Markdown，而是给 Markdown 加一个 Ctrl+F。**

## 五、初步分析（SKILL 设计方向）

### 需要解决的问题

1. **角色设定注入**：SKILL 如何让 Agent 理解"当前跑团的人设"并适配角色
2. **规则知识获取**：如何让 Agent 获取特定 TRPG 规则（CoC、DND、PF、赛博朋克红等）
3. **日志持久化**：跑团日志如何存储、如何快速检索
4. **场内外分离**：如何在输出中结构性地分隔场外分析和场内记录
5. **GM/Player 双模式**：如何切换服务模式
6. **🔴 长团可扩展性**：如何在数月长团/多副本战役中维持检索性能
7. **🔴 忠诚契约**（新）：PL 模式（存活+掘利+禁出卖）/ GM 模式（不无聊+能继续+戏剧化管理+剧本知识库）
8. **🔴 移动端兼容性**（新）：手机跑团基数大，Markdown 零依赖保障核心功能跨平台

### SKILL 结构最终状态

- 角色设定注入（背景描述 + 角色卡模板）
- 知识库管理（规则书摘要、日志归档）
- 工作流：车卡 / 六步日志 / 形势分析 / 防超游 / 平台切换 / rule_lib
- 输出格式约束（场外[OOC] vs 场内[IC] 标记）
- 🔴 忠诚契约：PL 模式六大职责 + GM 模式七大职责 + 戏剧化建议模板
- 🔴 可扩展存储层：Markdown（人类可读）+ SQLite（AI检索引擎）双写架构 + 自动分页
- 🔴 环境检测：check_env.py + 4 国内 PyPI 镜像 + pip 失败不影响核心工作流的降级设计
- 🔴 手机兼容：Markdown 100% 移动端可读，桌面端 Agent 执行写操作
